# Reference — Meta-rules governing vault design

Loaded when: you're designing vault-adjacent tooling, auditing existing
scripts, or debugging a symptom that feels structural. These six rules
live in `$VAULT/process/knowledge_base_design.md` and govern the
vault's multi-author safety properties.

They are the rules most likely to be violated by a fresh AI, and also
the rules whose violations are hardest to spot without the incident
context that produced them. Full text of each rule is in the vault;
this reference gives you the distilled form plus the failure mode.

---

## r1 — Scripts are authoritative over hand-written generated content

**When:** Any file that is regenerated by a script (index, manifest
stats block, script-version table, etc.).

**Rule:** Let the script write the file from day one. Never hand-write
content that a generator will overwrite.

**Why it exists:** During the V1–V5 vault build, `_INDEX.md` was
hand-written before `reindex.py` was introduced. The first dry-run
showed ~340 lines of cosmetic diff that made real drift impossible to
spot and required a separate sync commit. Had the script owned the
file from day one, the diff baseline would have been zero.

**Violating it looks like:** you hand-edit `_INDEX.md` to add a line
for a new file "because it's faster than running reindex", the next
`reindex.py` run overwrites your edit with a subtly different format,
and now the git diff is noise.

**How to tell you're applying it:** anything inside a
`<!-- SECTION:BEGIN -->` / `<!-- SECTION:END -->` sentinel block is
script-owned. Don't edit inside those markers. The sentinel block
pattern itself is r2 (not listed here) — it's the mechanism that makes
r1 safely coexist with hand-written prose in the same file.

**Citing incident:** V1–V5 vault build.

## r13 — Every script has a `SCRIPT_VERSION = N` integer, exposed via `--version`

**When:** Writing or updating any vault script (`validate.py`,
`reindex.py`, `search.py`, `stats.py`, `_vault.py`).

**Rule:** Every script defines `SCRIPT_VERSION = N` as an integer
module-level constant. Increment on any behavioral change. Expose via
`--version` CLI flag. `_BOOTSTRAP.md §9` maintains a
`<!-- SCRIPT_VERSIONS:BEGIN/END -->` sentinel block declaring the
minimum required version of each script, regenerated automatically by
`stats.py`.

**Why it exists:** Scripts evolve — new flags, bug fixes, behavioral
changes. An AI running a stale local copy of `search.py` would return
wrong results with no indication anything was wrong. The integer
version is simpler to compare than semver — the agent only needs
`mine >= minimum`.

**Violating it looks like:** you add a new flag to `search.py` but
forget to bump `SCRIPT_VERSION`, the bootstrap table doesn't update,
and a future AI running an old copy of the script doesn't know it's
missing the flag.

**How to tell you're applying it:** grep the file for `SCRIPT_VERSION`
and check the bootstrap table matches. `orient.sh` prints the versions
on startup so drift is visible from session one.

**Citing incident:** Vault script implementation session, 2026-04-14.

## r14 — Enumerations need a single source of truth, enforced by imports

**When:** A schema-driven system has a set of directories, types,
categories, or similar enumerations referenced in more than one script,
config file, or document.

**Rule:** Define the enumeration in exactly one place (a shared
constants module, a config file, a schema file) and require every
consumer to import or load it. Hand-rolled duplicates in individual
scripts are drift sites waiting to happen. When you discover a
duplicated enumeration, consolidate it **before** making any other
change — do not extend a duplicate "for now".

**Why it exists:** `_vault.py` exposed `TYPE_ORDER`/`CONTENT_DIRS`/
`TYPE_LABEL` as shared constants. `search.py` and `stats.py` imported
them; `validate.py` and `reindex.py` hand-rolled their own copies.
When a previous session added `agent-profile` to `_vault.py`, the fix
only reached half the scripts — `validate.py` silently skipped the
entire `agents/` directory and `reindex.py` never rendered an agent
subsection. A separate hardcoded regex inside `validate_index_parity`
(`(?:tools|techniques|...)/`) was a **third** drift site. Three drift
sites, all silent, all caused by the same root: enumerations duplicated
instead of imported.

**Violating it looks like:** `validate.py` reports `FULL_SUCCESS` while
completely skipping one content directory. A new content type is added
to the schema but some scripts don't know about it. A regex over
directory names doesn't match a newly-added directory.

**How to tell you're applying it:** in any vault script, search for
hardcoded directory name lists or type name lists. If they exist and
aren't imported from `_vault.py`, you have a drift site. Fix by
importing.

**Citing incident:** `2026-04-14-fresh-ai-walkthrough-fixes.md`
(agents/ orphan).

## r15 — Silent-skip iterators reward incorrect output — prefer loud-fail on unknowns

**When:** Writing any iterator, validator, linter, or indexer that
walks a set of directories/types/categories.

**Rule:** Do not iterate a hardcoded whitelist and stop there. **Also
list the parent location and warn (or fail) on any item outside the
whitelist.** An iterator that silently skips unknown items reports
success when a user creates something the system doesn't know how to
handle — rewarding incorrect output with a green checkmark.
**Whitelist iteration is correct; whitelist-only iteration is a bug
class.**

**Why it exists:** `validate.py` reported `FULL_SUCCESS (33/33 files
checked)` while completely skipping the `agents/` directory — which
contained a file written per the documented contribution protocol.
A fresh AI following `_WRITE_PROTOCOL.md` would write a profile, run
the validator, see green, and have no indication its file was ignored.
A `VERIFY_WARN: unknown content directory 'agents/'` on the first run
after the directory was created would have surfaced the gap
immediately.

**Violating it looks like:** you add a new `skills/` subdirectory to
the vault, write files into it, run the validator, get green, and
months later discover nothing in `skills/` has ever been validated or
indexed. The same pattern repeats for `comms/` being orphaned after
`agents/` was fixed — see r17.

**How to tell you're applying it:** the iterator has two phases —
first, enumerate siblings and warn on any outside the whitelist;
second, iterate the whitelist. If it only has phase two, it's a
silent-skip iterator. The `check_unknown_dirs` UNIT in `validate.py`
is the reference implementation.

**Citing incident:** `2026-04-14-fresh-ai-walkthrough-fixes.md`.

## r16 — Static-regex introspection cannot see imported constants

**When:** Writing a tool that greps or regex-searches other source
files for a named constant (e.g., `SCRIPT_VERSION = N`, `VERSION = "..."`)
rather than importing and reading them at runtime.

**Rule:** The constant must be **defined locally** in every target
file the introspector scans. A regex has no runtime knowledge of
imports — it only sees the text on disk. Consumers that would prefer
to import the value to avoid duplication must instead define their
own local copy. Document the duplication requirement near the constant.

**Why it exists:** `search.py` initially imported `SCRIPT_VERSION` from
`_vault.py` to avoid duplication. `stats.py`'s `get_script_versions()`
uses a static regex to build the `_BOOTSTRAP.md` SCRIPT_VERSIONS table,
and the regex saw no local `SCRIPT_VERSION = N` line in `search.py` —
so the bootstrap table printed `| search.py | missing | ...`. The fix
(define the constant locally in every target) is trivial; the failure
mode is easy to repeat.

**Violating it looks like:** bootstrap table shows `missing` for a
script you know exists. Or a license-header linter fails on a file
with a correctly imported license constant.

**How to tell you're applying it:** any script the version-table
regex is supposed to find has `SCRIPT_VERSION = N` as a literal line
visible to text search. Not `from _vault import SCRIPT_VERSION`.

**Citing incident:** `2026-04-14-vault-script-implementation.md`.

## r17 — Promoted rules must be retrofitted into existing tooling in the same session

**When:** Promoting a new rule that prescribes behavior for existing
tools, validators, indexers, or scripts in the vault (or any system
the rule applies to).

**Rule:** A rule is not "done" when it is written. In the **same
session** that promotes a new rule, retrofit every existing consumer
the rule applies to. If the rule says "iterators must warn on unknown
items", every iterator in the codebase gets the warning. If the rule
says "all schema files must be validated", the validator gets the
schema check. Documenting a rule without applying it to current state
produces "dead code rules" — they live in the vault, get cited in
incident reports, and silently fail to prevent the next instance of
the bug they were written to stop.

**Why it exists:** During the fresh-AI walkthrough fix session of
2026-04-14, `r15` was promoted — the silent-skip iterator rule. The
session then closed without retrofitting `validate.py` to actually do
loud-fail. Hours later, a second session added the `memo` entry type
to the schema, created `comms/`, and wrote a memo. The vault scripts
had not been updated for the new type, and `validate.py` reported
`FULL_SUCCESS (36/36 files checked)` while completely skipping the
`comms/` directory. **This is the exact failure mode r15 was written
to prevent — but r15 had only ever been described, never implemented.**

**Violating it looks like:** you write a rule promoting a new
invariant, commit the rule to the vault, and the very next write
violates it because the tooling doesn't enforce it yet. The rule
exists but is functionally decorative.

**How to tell you're applying it:**

Same-session retrofit checklist:
1. Write the rule (the easy part)
2. Grep the vault/codebase for all sites the rule applies to
3. Update each site to satisfy the rule
4. Add a regression test or adversarial check that proves the retrofit
5. Run handoff.sh — validation should catch any miss
6. Only then mark the rule promotion task complete

**Citing incident:**
`2026-04-14-second-fresh-ai-walkthrough-memo-orphan.md`.

## How these rules compose

These six rules are not independent — they form a consistency layer:

- **r1** is about separation of concerns between hand-written and
  generated content. It requires the sentinel block pattern to work.
- **r13** applies r1 to script version tracking specifically, because
  versions are the mechanism that makes cross-session safety work.
- **r14** is about data: enumerations must have exactly one definition.
- **r15** is about control flow: iterators must handle the unknown
  case, not just the known case.
- **r16** is a constraint on r14: imports solve duplication at runtime
  but not at regex-scan time. Introspective tooling must account for
  this.
- **r17** is the meta-rule that binds all of the above to session
  discipline: rules don't enforce themselves, so promoting one
  without applying it to existing tooling just defers the next
  incident.

When designing vault tooling, walk through all six and ask how the new
code interacts with each. If any one of them is unsatisfied, you have
a future incident with a foreknown root cause.
